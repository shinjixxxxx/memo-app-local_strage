# メモアプリ Google Drive 同期機能 プラン

## Context（なぜ作るか）
現在 `memo-app_free-check-seet_02.html` のデータは **localStorage のみ**に保存されており、PC・スマホ・iPad 間でデータを共有できない。一人のユーザーが複数端末を時間差で使う運用において、メモを端末間で同期したい。

経路としては Google Drive をハブにする。ユーザーは「Googleにログイン済み」が前提で、一般ユーザーでも使えること（GitHub等の専門知識を要しない）を重視する。データの取りこぼしを絶対に避けるため、競合解決は「勝ち負け判定をせず常に和集合で全部残す」方針とする。

---

## 確定した設計判断（相談で合意済み）

| 項目 | 決定 |
|---|---|
| 保存先 | Google Drive（`appDataFolder` 隠し領域・平文JSON） |
| 認証 | 初回のみ OAuth 同意、以降は silent 取得で自動。**完全無認証は不可**（Google仕様） |
| 同期サイクル | **読み込み → マージ → 保存** の1本。手動/自動/起動時すべて同じ処理を共用 |
| 同期トリガー | ①起動(アクセス)時の自動 ②定期(分単位・初期30分) ③設定ダイアログ内「同期する」手動 |
| オフライン | 同期しない（編集は従来通り完全オフライン動作） |
| マージ：ページ | **ID基準**で全部残す。競合は両方残す |
| マージ：メモ行 | テキストの**和集合・重複排除**。行が増える方向（消さない） |
| ID衝突対策 | 端末ID(UUID)をページIDに組み込む（別端末の同時生成でも衝突しない） |
| 端末識別 | 端末ID + 端末名。端末名は初回 userAgent から種別(Mac/iPhone/iPad)を推測し初期値、手修正可。**ページ単位**で由来追跡 |
| 設定UI | 既存フロートUIの設定ダイアログ `#ps-settings-overlay` 内に集約 |
| 秘匿性 | 平文JSONでOK（Driveのアカウント制御に委ねる） |
| GitHub | 不採用（一般ユーザー向け） |
| 外部依存 | 普段はゼロ。同期時のみ GIS スクリプトを動的ロード |

## 現状コードの把握（変更の起点）

対象ファイル：`memo-app_free-check-seet_02.html`（単一HTML・`<script>` 内 DOMContentLoaded クロージャ）

**localStorage 構造**
- `fs02_pages` = `[{id, name}]`（先頭が base ページ）
- `fs02_active` = アクティブページID
- ページ別キー `fs02_<id>_<suf>`：
  - `cb` = チェックボックス真偽の配列（JSON）
  - `memo` = メモ文字列の配列（JSON、上部textarea/下部textarea/各行input を含む順序依存）
  - `rows` = 行数
  - `date` = 日付文字列(yyyy-mm-dd)
- `fs02_title` = ドキュメントタイトル

**関係する既存関数**（再利用・拡張対象）
- `newId()` … ページID生成（時刻+乱数）→ **UUID+端末ID方式に変更**
- `getPages()/setPages()/getActive()/setActive()` … ページ配列の出入口
- `savePage(id)/saveAll()/loadPage(id)` … 保存・復元
- `openSettings()/closeSettings()/saveSettings()` … 設定ダイアログ制御（`#ps-settings-overlay`）
- 設定ダイアログDOM：現在は `#ps-page-title` 入力＋ `#ps-settings-close`/`#ps-settings-save` のみ

## フェーズ構成

### フェーズ0：PoC（技術検証）※本実装の前に必須
本体には手を入れず、**検証専用の使い捨てHTML 1枚**（例 `poc-drive.html`）で確認する。`npm run server`（localhost:2001）経由で開く（`file://` ではGISが動かない）。

**事前準備（手順書／発行作業はユーザーが実施）**
1. Google Cloud Console でプロジェクト作成
2. 「APIとサービス」→ Google Drive API を有効化
3. OAuth 同意画面を構成（External / テストユーザーに自分のGmailを追加）
4. 認証情報 → OAuthクライアントID（種別：ウェブアプリ）を発行
5. **承認済みJavaScript生成元**に `http://localhost:2001` を登録
6. 発行された **クライアントID** を PoC HTML に貼る

**検証項目（重要度順）**

| # | 確認 | 判定 |
|---|---|---|
| ① | GIS で OAuth 同意 → アクセストークン取得 | 連携可否 |
| ② | トークンで `appDataFolder` に JSON 1件書き込み | 書込可否 |
| ③ | リロード/別タブで読み戻せる | 永続性 |
| ④ | **2回目以降、同意画面なしで silent トークン取得**（`prompt: ''`） | ★本命：初回のみ同意が成立するか |
| ⑤ | トークン失効時に再同意でリカバリ | 運用継続性 |

**合格ライン**：①〜④クリアで本実装へ進む。④が不可なら体験設計を見直す分岐。

---

### フェーズ1：データのシリアライズ基盤（Drive非依存）
- localStorage 全体（`fs02_*` と `fs02_title`）を**1つの正規化JSONドキュメント**に集約/復元する関数を新設：
  - `exportData()` … 全ページ＋各ページの cb/memo/rows/date を1オブジェクトに
  - `importData(obj, {merge})` … 取り込み（merge=false で置換、true でマージ）
- ドキュメント形（案）：
  ```json
  {
    "schema": 1,
    "device": {"id": "<uuid>", "name": "メインMac"},
    "title": "...",
    "pages": [
      {"id":"...", "name":"...", "origin":"<端末id>",
       "rows": 6, "date":"2026-06-07",
       "memos": ["...", "..."], "checks": [true,false,...]}
    ]
  }
  ```
- ローカルファイルへ **エクスポート/インポート**（Blob ダウンロード / `<input type=file>` 読込）。オフライン可。

### フェーズ2：端末ID・端末名
- 起動時 `fs02_device`（`{id, name}`）が無ければ生成：
  - `id = crypto.randomUUID()`
  - `name` 初期値 = userAgent から `Mac` / `iPhone` / `iPad` / `その他` を推測
- `newId()` を `端末ID由来のプレフィックス + UUID` 方式に変更（既存 `base`・既存IDは温存、新規のみ新方式）
- ページ生成時に `origin = 端末ID` を記録 → ページ単位で由来追跡

### フェーズ3：設定ダイアログ拡張
`#ps-settings-overlay` 内に追加（既存のティールUIスタイルを踏襲）：
- 端末名 入力欄
- 同期間隔（分）数値入力（初期30）
- **[Drive連携]** ボタン（初回OAuth同意。連携状態の表示も）
- **[同期する]** ボタン（読み込み→マージ→保存を即実行）
- 同期状態表示（「未連携」「同期中…」「最終同期：◯分前」）

### フェーズ4：マージエンジン（Drive非依存・単体テスト可能）
- `mergeDocs(localDoc, remoteDoc)` を純粋関数として実装：
  - ページ：ID基準で突き合わせ。両方にあれば中身マージ、片方だけなら追加（全部残す）
  - メモ行：両ページの memos を結合 → trim 後の空文字除外 → 重複排除（和集合）。行が増える方向
  - チェック状態：和集合で行が増えるため、行とチェックの対応は「テキスト→チェックの論理和」等の単純規則で寄せる（詳細は実装時に確定、安全側＝消さない）
- 純粋関数なので Node で単体テストしやすい

### フェーズ5：Drive同期統合
- GIS スクリプトを**同期実行時に動的ロード**（未連携/オフライン時はロードしない）
- `syncNow()` ＝ ①Drive読込 → ②`mergeDocs` → ③ローカル反映＆保存 → ④Driveへ保存
- トリガー配線：起動時（オンライン時のみ）／`setInterval`（分単位）／「同期する」ボタン
- オフライン検出（`navigator.onLine`）で自動同期スキップ

## 検証方法
- **PoC**：localhost:2001 で `poc-drive.html` を開き①〜⑤を手動確認
- **マージ**：`mergeDocs` を Node で単体テスト（端末A/B相当の2ドキュメントを入力し、和集合・重複排除・全ページ残存を確認）
- **E2E**：2つのブラウザプロファイル（または localStorage を手で差し替え）で別端末を擬似再現 → 同期して両者の和集合になることを puppeteer で確認
- **回帰**：既存のオフライン編集・localStorage保存・opnボタン等が従来通り動くこと

## 実装順序（推奨）
1. **フェーズ0（PoC）** を最優先。④ silent 取得の成否で全体の体験設計が決まる
2. PoC通過後、フェーズ1（シリアライズ）→ フェーズ2（端末ID）→ フェーズ4（マージ純粋関数＋単体テスト）
3. フェーズ3（設定UI）→ フェーズ5（Drive統合）で結線
4. 各フェーズごとに既存機能の回帰確認
