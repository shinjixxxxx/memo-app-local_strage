# メモアプリ 同期機能 プラン（iCloud Drive ＋ ファイル手動同期）

## Context（なぜ作るか）
`memo-app_free-check-seet_02.html` のデータは現在 **localStorage のみ**で、Mac・iPad・iPhone 間で共有できない。一人のユーザーが3台のApple端末を時間差で使う運用で、メモを端末間で同期したい。

当初は Google Drive API での自動同期を検討したが、**OAuthクライアントID発行など初期設定が複雑で「普通の人も使える」目的と矛盾**するため取りやめた。

最終方針は **「アプリはローカルの JSON ファイルを読み書きするだけ。その置き場所を iCloud Drive 同期フォルダにして、Apple に裏で3台へ配ってもらう」**。アプリ自体はクラウドを一切知らない。OAuth・APIキー・外部依存ゼロ。

データの取りこぼしを避けるため、競合解決は**勝ち負け判定をせず常に和集合で全部残す**。

## 確定した設計判断（相談で合意済み）

| 項目 | 決定 |
|---|---|
| 対象端末 | **Apple3台（Mac・iPad・iPhone）**。Windows/Android は対象外 |
| 保存先 | **iCloud Drive の同期フォルダ**。アプリはクラウドを意識しない（ただのローカルファイル） |
| アプリの役割 | JSONファイルを**読む/書く**だけ。端末間配布は iCloud が自動で行う |
| ファイル | **1個・固定名 `memo-sync.json`・上書き運用** |
| 読み込み | 全端末OK。読み込んだら**常にマージ**（和集合・全部残す・重複排除）→ マージ後に書き出し |
| 強制操作 | マージ専用だと**削除できずメモが増え続ける**問題への対処。**強制書出**=この端末でファイルを上書き／**強制読込**=ファイルでこの端末を上書き（どちらもマージなし・確認ダイアログ必須） |
| 書き出し（Mac） | **File System Access API** で同じファイルに本当に上書き（Chrome/Edge。Safari不可） |
| 書き出し（iPad/iPhone） | ダウンロード → 「ファイル」アプリで iCloud に**手で上書き**（`(1)`が付くので置換） |
| Mac のファイル指定 | **ファイル指定方式**：初回に `memo-sync.json` を1回選ぶ → 以降そのハンドルで読み書き |
| 閉じる時の同期 | 未同期の変更があれば**閉じる時に警告ダイアログ**（全端末）＋**Macは自動上書きを試みる**（動けば採用）＋未同期フラグで次回起動時に通知 |
| マージ：ページ | **ID基準**で全部残す。競合は両方残す |
| マージ：メモ行 | テキストの**和集合・重複排除**。行が増える方向（消さない） |
| ID衝突対策 | 端末ID(UUID)をページIDに組み込む |
| 端末識別 | 端末ID + 端末名。端末名は初回 userAgent から種別(Mac/iPhone/iPad)を推測し初期値、手修正可。**ページ単位**で由来追跡 |
| 設定UI | 既存フロートUIの設定ダイアログ `#ps-settings-overlay` 内に集約 |
| 秘匿性 | 平文JSONでOK |
| OAuth/APIキー/外部依存 | **すべて不要** |

## ブラウザ制約（重要・設計の前提）
| 操作 | Mac (Chrome/Edge) | iPad/iPhone (Safari) |
|---|---|---|
| 読み込み（ファイルを開く） | ✅ | ✅（`<input type=file>` で「ファイル」アプリから選択。DL不要） |
| 書き出し（更新） | ✅ 同一ファイルに上書き(FSA API) | ⚠️ ダウンロードのみ。同名は `(1)` が付く→手で上書き |
| 閉じる時の自動書き出し | △ FSA なら可能性あり（要検証） | ❌ 不可（beforeunloadでのDL禁止） |

- `beforeunload` で**ファイル保存ダイアログ/DLを出すことは禁止**。よって「閉じる時の確実な自動書き出し」は不可。代替として**警告ダイアログ＋未同期フラグ**で守る。
- Safari は File System Access API 非対応（Appleの方針）。回避策なし。

## フェーズ構成

### フェーズ1：シリアライズ基盤（Drive非依存）
- `exportData()` … localStorage 全体（`fs02_*` + `fs02_title`）を1つの正規化JSONへ
- `importData(obj)` … 取り込み（常にマージ）
- ドキュメント形（案）：
  ```json
  {
    "schema": 1,
    "device": {"id": "<uuid>", "name": "メインMac"},
    "title": "...",
    "pages": [
      {"id":"...", "name":"...", "origin":"<端末id>",
       "rows": 6, "date":"2026-06-07",
       "memos": ["...", "..."], "checks": [true,false,...]}
    ]
  }
  ```

### フェーズ2：端末ID・端末名
- 起動時 `fs02_device`（`{id, name}`）が無ければ生成：`id = crypto.randomUUID()`、`name` 初期値 = userAgent から `Mac`/`iPhone`/`iPad`/`その他` を推測
- `newId()` を `端末ID由来プレフィックス + UUID` に変更（既存 `base`・既存IDは温存、新規のみ新方式）
- ページ生成時に `origin = 端末ID` を記録 → ページ単位で由来追跡

### フェーズ3：マージエンジン（純粋関数・単体テスト可能）
- `mergeDocs(localDoc, remoteDoc)`：
  - ページ：ID基準で突き合わせ。両方にあれば中身マージ、片方だけなら追加（全部残す）
  - メモ行：両ページの memos を結合 → trim 後の空文字除外 → 重複排除（和集合）。行が増える方向
  - チェック状態：行が増えるため「テキスト→チェックの論理和」等の単純規則で寄せる（安全側＝消さない。詳細は実装時確定）
- Node で単体テストしやすい

### フェーズ4：同期I/O（端末別の読み書き）
- **読み込み（全端末共通）**：`<input type="file">` で `memo-sync.json` を選択 → パース → `importData`（マージ）→ ローカル反映＆保存 → **続けて書き出し**
- **書き出し（Mac / FSA）**：初回に `showSaveFilePicker`/`showOpenFilePicker` で `memo-sync.json` のファイルハンドルを取得・記憶 → 以降は同ハンドルに `createWritable()` で上書き
- **書き出し（iPad/iPhone）**：Blob を `a[download]` でダウンロード → ユーザーが「ファイル」アプリで iCloud に上書き
- 起動時に capability 判定（`window.showOpenFilePicker` 有無）でMac経路/Safari経路を出し分け

### フェーズ5：設定ダイアログ＆「閉じる時」対応
- `#ps-settings-overlay` に追加（既存ティールUI踏襲）：
  - 端末名 入力欄
  - **[同期]** ボタン（読み込み→マージ→ローカル反映→書き出し の一連）
  - （Mac）**[ファイル指定]** ボタン（`memo-sync.json` のハンドルを選ぶ／変更）
  - 同期状態表示（「最終同期：◯分前」「未同期の変更あり」など）
- **未同期フラグ**：メモ編集時に「未書き出し変更あり」を立て、同期完了で下ろす
- **`beforeunload`**：未同期フラグが立っていれば警告ダイアログ。**Macのみ** FSA で自動上書きを試みる（成功すれば理想・失敗しても警告で担保）
- **次回起動時**：未同期フラグが残っていれば「前回分が未同期です。同期しますか？」と通知

## 検証方法
- **マージ**：`mergeDocs` を Node で単体テスト（端末A/B相当の2ドキュメント → 和集合・重複排除・全ページ残存を確認）
- **Mac**：Chromeで `memo-sync.json` を指定 → 編集 → 同期で同一ファイルが上書きされることを確認。閉じる時の自動上書き挙動を実機確認
- **iPad/iPhone**：Safari実機で「ファイル」アプリから読み込み → 編集 → ダウンロード → iCloudへ手で上書き、の一連を確認
- **端末間**：Mac で書き出し → iCloud 同期 → iPad で読み込みマージ → iPad で追記 → 書き出し → Mac で読み込みマージ、と一周して**両方のメモが残る**ことを確認
- **回帰**：既存のオフライン編集・localStorage保存・opnボタン等が従来通り動くこと

## 強制書出 / 強制読込（マージしない上書き）
マージ専用（和集合）には**削除の概念が無く、メモが増え続ける**（消してもファイルから復活する）。これを解消する escape hatch。設定ダイアログに「強制（マージしない・上書き）」セクションとして配置、いずれも**確認ダイアログ必須**。

| 操作 | 動作 | 使う場面 | 失われるもの |
|---|---|---|---|
| **同期する** | 読込→マージ→書込（足し算） | 普段。安全だが消せない | なし |
| **強制書出** | この端末の内容で**ファイルを上書き**（読込・マージなし） | 整理（削除）を全体に反映 | ファイル側の未取り込み分 |
| **強制読込** | ファイルの内容で**この端末を上書き**（マージなし・置換） | ファイルを正として丸ごと採用／端末リセット | この端末だけのローカル変更 |

- **実装**：強制読込は `applyDoc(remote)`（マージなし置換）を使い、ファイルに無いページの孤立 localStorage キー（`fs02_<id>_*`）も掃除。強制書出は `writeHandle`/`downloadDoc` で local の `exportData()` をそのまま書くだけ。
- **削除を全端末へ伝える手順（重要）**：①1台で不要メモを削除→**強制書出**（ファイルがその端末の内容になる）②他端末は**強制読込**（ファイルを丸ごと採用）。
  - ⚠️ ここで他端末が「同期する」(マージ)を押すと、ローカルに残る削除済みメモがファイルに**復活**する。削除整理の直後は各端末で「強制読込」する運用。

## 実装で確定した仕様
- **チェック状態のマージ**：同一テキスト行が両端末にある場合、チェック（予定/作業/終了/確認の4個）は**インデックスごとの論理和（OR）**で寄せる（安全側＝消さない）。行は和集合で増える方向。
- **閉じる時の自動上書き（Mac）**：`beforeunload` 内の非同期書き込みは完了保証がないため、**`visibilitychange`（hidden）で書き込む方式**を採用。さらに**無操作で書ける（queryPermission==='granted'）場合のみ**実行し、権限プロンプトは出さない。`beforeunload` は未同期時の**警告ダイアログのみ**（全端末で確実に動く担保）。
- **前回未同期の通知**：起動時、未同期フラグが残りかつ「同期を使っている人」（過去に同期した／ファイル指定済み）に限り「今すぐ同期しますか？」と通知（非利用者は煩わせない）。
- **自動同期タイマー**：Mac(FSA)・オンライン・無操作で書ける場合のみ静かに実行（既定30分・設定で変更可）。
- **ファイルハンドルの永続化**：選んだ `memo-sync.json` のハンドルは **IndexedDB** に保存し、リロードをまたいで再利用。

## 留意点（残課題）
- iPad/iPhone のダウンロードファイル名が `(1)` で増える点は手動上書きで対処（仕様として許容）
- FSA の実ファイル上書きと `visibilitychange` 自動書き込みは Mac 実機での最終確認が必要（ロジック・マージ純粋関数は検証済み）

## 実装順序（推奨）
1. フェーズ1（シリアライズ）→ フェーズ2（端末ID）→ フェーズ3（マージ純粋関数＋単体テスト）
2. フェーズ4（読み書きI/O・端末別）
3. フェーズ5（設定UI・未同期フラグ・閉じる時対応）
4. 各段階で既存機能の回帰確認、最後に端末間の一周テスト
